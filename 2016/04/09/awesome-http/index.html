Awesome-HTTP

HTTP协议解决的问题：

  1. 资源：从何获取( URI –> URL and URN )，资源类型( MIME )等（一个页面是多个资源的集合）
  2. 事务：一个事务＝一个请求＋一个响应，请求方法，状态码＋状态码解释等（一个页面需要执行多个事务）
  3. 报文：请求报文＋响应报文，起始行、首部（空行结束）、主体
  4. 连接：端口号＋TCP/IP，端口号、IP地址（或者域名）由URL提供
  5. Web的结构组件：客户端，服务端（重要的两个），代理（安全、应用集成、性能优化），缓存代理（性能优化），网关（协议转换），隧道（盲发），Agent代理（例如浏览器、网络蜘蛛，这是客户端的程序）等等

1. URI 与 资源

URI的两个子集

URL（位置标识资源） 和 URN（名称标识资源）

URL

协议＋服务器位置（包括端口号）＋资源路径

完整语法：

<scheme>://<user>:<password>@<host>:<port>/<path>;<params>?<query>#<frag>

相对URL、浏览器自动扩展URL等都是URL的快捷方式

URL的编码问题：

  1. URL使用的是US-ASCII字符集，采用7位二进制码；
  2. 超过US-ASCII字符集范围的字符将使用“转义”表示法进行转义；（%加上两位16进制数表示一个字符）
  3. 保留关键字符不能拿来使用

推荐所有URL在发送请求前都对不安全的字符进行编码转义

URL中主要的协议（或称方案）：

  1. http （除了没有用户名和密码之外，与通用的URL格式相符。如果省略了端口，就默认为80）
  2. https （语法与HTTP的语法相同，默认端口为443。唯一的区别在于方案https使用了网景的SSL，SSL为HTTP连接提供了端到端的加密机制。）
  3. ftp、file……

URL的不完美：

如果资源被移走了， URL 也就不再有效了

完美的方案 –> URN

只要有了对象的准确名称，则不论其位于何处都可以找到这个对象

URN的一种实现：PURL（永久资源定位符）

基本思想是在搜索资源的过程中引入另一个中间层，通过一个中间资源定位符服务器对资源的实际 URL 进行登记和跟踪。客户端可以向定位符请求一个永久 URL，定位符可以以一个资源作为响应，将客户端重定向到资源当前实际的 URL 上去

2. HTTP报文

  1. 报文流：用来描述报文的传输过程以及方向，把一个事务看成是一条河流，那么上游，下游，流入，流出就很好理解了
  2. 方法：GET POST PUT DELETE等，还有一些服务器自有的扩展方法！（请求报文起始行）
  3. 状态码：除了规定的，可以按规范中的范围意义自定义（响应报文起始行）
  4. 原因短语：仅仅是解释状态码（响应报文起始行）
  5. 版本号：比如，HTTP/2.22 就比 HTTP/2.3 的版本要高，因为 22 比 3 大
  6. 首部（将长的首部行分为多行可以提高可读性，多出来的每行前面至少要有一个空格或制表符（tab）。后面细讲）
    7.主体部分：很多类型数据的源数据

2.1 方法

方法只是用来告诉开发者应该做什么，而不是具体实现了什么

安全方法：GET HEAD（遵循 HTTP/1.1 规范就必须实现）

理解：PUT POST TRACE OPTIONS

  • TRACE：用于验证请求是否如愿穿过了请求 / 响应链，可以用来查看代理和其他应用程序对用户请求所产生效果

  • OPTIONS：请求 Web 服务器告知其支持的哪些功能（包括请求方法）

扩展方法（规范中未定义的）

2.2 首部

1. 通用首部
  1. 信息类（略）
  2. 缓存首部：Cache-Control
2. 请求首部
  1. 信息类（略）
  2. Accept类首部：告知服务器它们想要什么，可以使用什么，以及最重要的，它们不想要什么（Accept Accept-Charset Accept-Encoding Accept-Language TE）
  3. 条件请求首部：为请求加上某些限制
  4. 安全请求首部：对请求进行质询 / 响应认证（Authorization Cookie Cookie2）
  5. 代理请求首部（Max-Forward Proxy-Authorization Proxy-Connection）

某些重要的首部会详细讨论

3. 响应首部
  1. 协商首部（Accept-Ranges Vary）
  2. 安全响应首部：HTTP 的质询 / 响应认证机制的响应侧
4. 实体首部
提供了有关实体及其内容的大量信息，从有关对象类型的信息，到能够对资源使用的各种有效的请求方法
  1. 信息类（略）
  2. 内容首部（Content-*）
  3. 实体缓存首部（ETag Expires Last-Modified）

3. 连接管理

性能优化点：

  1. Connection首部指定不传输哪些首部

  2. 显示写定图片尺寸

  3. 并行连接

  4. 持久连接
  5. 管道化连接
  6. 复用的连接

并行连接：

同时（每个连接之间有软件延迟）打开多个HTTP连接

优点：

  1. 网络带宽够用、内存够大、服务器性能好的情况下，会加快加载速度；
  2. 多个事务在执行，让用户感觉速度还行

缺点：

  1. 带宽不够时，由于事务之间的竞争，会让每个事务执行变慢；
  2. 内存不够时，电脑性能受影响；
  3. 服务器在并行数量累积情况下，难以承受，会更慢；
  4. TCP的建立与慢启动的缓慢过程仍没改变

中庸：

客户端将并行连接的总数限制为一个较小的值（通常是 4 个）；服务器可以随意关闭来自特定客户端的超量连接

持久连接：
建立一次TCP连接，传输多次数据，要避免出现空闲未使用情况
